# ОПИСАНИЕ ПРОГРАММЫ

## ОСНОВНАЯ ЧАСТЬ

### 1. Общие сведения

**1.1. Обозначение и наименование программы**

Обозначение: AgroVzor  
Наименование: Экосистема агротехники в поле для учета и мониторинга зерна

**1.2. Программное обеспечение, необходимое для функционирования программы**

Для серверной части (AgroVzorBack):
- Java Runtime Environment версии 17 или выше
- Spring Boot Framework версии 4.0.0
- Spring Data JPA для работы с базой данных
- PostgreSQL версии 16 или выше (или совместимая версия)
- Docker и Docker Compose (для контейнеризации)
- Gradle версии 8.4 или выше (для сборки проекта)

Для клиентской части (AgroVzorFront):
- Node.js версии 20 или выше
- Next.js Framework версии 16.0.3
- React версии 19.2.0
- TypeScript версии 5 или выше
- Современный веб-браузер с поддержкой ES6+ (Chrome, Firefox, Edge, Safari)

Дополнительные библиотеки и зависимости:
- SpringDoc OpenAPI 3 (Swagger) версии 2.3.0 для документации API
- Lombok для упрощения кода
- PostgreSQL JDBC драйвер
- Recharts версии 3.5.0 для визуализации данных
- Leaflet версии 1.9.4 для работы с картами
- Radix UI компоненты для пользовательского интерфейса

**1.3. Языки программирования, на которых написана программа**

- Java 17 (серверная часть)
- TypeScript 5 (клиентская часть)
- SQL (запросы к базе данных через JPA/Hibernate)

### 2. Функциональное назначение

**2.1. Классы решаемых задач**

Программа предназначена для автоматизации и мониторинга процесса трехзвенной уборки урожая зерновых культур. Система решает следующие классы задач:

1. **Управление организационной структурой:**
   - Учет сельскохозяйственных предприятий (компаний)
   - Управление сотрудниками и их допусками к технике
   - Учет парка сельскохозяйственной техники (типы, модели, единицы техники)

2. **Планирование и учет полевых работ:**
   - Регистрация работ на полях с привязкой к культурам и сортам
   - Учет рабочих выездов механизаторов
   - Планирование состава команды для выполнения работ

3. **Сбор и обработка телеметрических данных:**
   - Прием регулярных пакетов данных с бортовых модулей техники
   - Фиксация координат, веса зерна, уровня заполненности бункеров
   - Регистрация событий вызова и перегрузки зерна

4. **Аналитика и мониторинг:**
   - Расчет потерь зерна при уборке урожая
   - Вибродиагностика техники с детекцией аномалий
   - Мониторинг уровня топлива с выявлением подозрительных случаев кражи
   - Расчет индекса эффективности работы механизаторов
   - Автоматический подбор оптимальной команды для полевых работ

5. **Визуализация и отчетность:**
   - Отображение данных на интерактивных картах
   - Построение графиков и диаграмм аналитических показателей
   - Формирование отчетов для анализа работы техники и персонала

**2.2. Назначение программы**

Программа обеспечивает автоматизацию взаимодействия основных участников трехзвенной уборки урожая (комбайн, бункер-перегрузчик, зерновоз) и предоставляет инструменты для:
- Снижения потерь зерна на 15-30% за счет автоматического учета и контроля
- Оптимизации логистики и сокращения простоев техники
- Предотвращения краж топлива и зерна
- Повышения эффективности работы механизаторов
- Соответствия требованиям федерального законодательства (ФЗ №499-ФЗ «О зерне»)

**2.3. Функциональные ограничения на применение**

1. Программа требует наличия стабильного интернет-соединения для синхронизации данных между бортовыми модулями и сервером (при отсутствии связи данные сохраняются локально и передаются при восстановлении связи)

2. Для полноценной работы необходимо наличие бортовых аппаратных модулей, установленных на технику, с датчиками веса, уровня заполненности, вибрации и топлива

3. Система предназначена для работы с зерновыми, масличными и другими сыпучими культурами

4. Требуется наличие базы данных PostgreSQL для хранения информации

5. Клиентская часть работает в современных веб-браузерах с поддержкой JavaScript ES6+

### 3. Описание логической структуры

**3.1. Алгоритм программы**

Программа реализует клиент-серверную архитектуру с разделением на серверную (backend) и клиентскую (frontend) части.

**Основной алгоритм работы серверной части:**

1. **Инициализация приложения:**
   - Загрузка конфигурации из application.properties
   - Подключение к базе данных PostgreSQL
   - Инициализация Spring Boot контекста
   - Автоматическое создание/обновление схемы базы данных через Hibernate

2. **Обработка HTTP-запросов:**
   - Получение REST API запросов через контроллеры
   - Валидация входных данных
   - Вызов соответствующих сервисов для обработки бизнес-логики
   - Преобразование сущностей в DTO (Data Transfer Objects)
   - Возврат JSON-ответов клиенту

3. **Обработка телеметрических данных:**
   - Прием регулярных пакетов данных от бортовых модулей
   - Сохранение данных в базу данных
   - Анализ данных на предмет аномалий (вибрация, топливо)
   - Расчет аналитических показателей (потери зерна, эффективность)

4. **Аналитические расчеты:**
   - Расчет потерь зерна на основе данных о весе и уровне заполненности
   - Анализ вибрации с выявлением аномалий (порог > 100)
   - Детекция кражи топлива (падение уровня > 10% за короткий период)
   - Расчет индекса эффективности оператора (взвешенная сумма показателей движения, времени реакции и эффективности выгрузки)

**Основной алгоритм работы клиентской части:**

1. **Инициализация приложения:**
   - Загрузка React-компонентов
   - Проверка доступности backend API
   - Загрузка начальных данных для отображения

2. **Работа с данными:**
   - Отправка HTTP-запросов к REST API
   - Обработка ответов и преобразование в типизированные объекты TypeScript
   - Кэширование данных в состоянии компонентов
   - Обработка ошибок и fallback на демонстрационные данные при недоступности backend

3. **Визуализация:**
   - Отображение данных в таблицах, карточках и графиках
   - Построение интерактивных графиков с использованием Recharts
   - Отображение карт с координатами техники через Leaflet
   - Обновление данных в реальном времени

**3.2. Используемые методы**

**Серверная часть:**

1. **Архитектурные паттерны:**
   - MVC (Model-View-Controller) - разделение на контроллеры, сервисы и репозитории
   - Repository Pattern - абстракция доступа к данным через Spring Data JPA
   - DTO Pattern - передача данных между слоями через Data Transfer Objects
   - Mapper Pattern - преобразование между сущностями и DTO

2. **Методы обработки данных:**
   - CRUD операции через Spring Data JPA репозитории
   - Транзакционное управление через @Transactional
   - Валидация данных через Bean Validation
   - Обработка исключений через глобальные обработчики

3. **Методы аналитических расчетов:**
   - Статистический анализ для расчета потерь зерна
   - Пороговый анализ для детекции аномалий вибрации
   - Сравнительный анализ для выявления кражи топлива
   - Взвешенное усреднение для расчета индекса эффективности

**Клиентская часть:**

1. **Методы работы с состоянием:**
   - React Hooks (useState, useEffect) для управления состоянием
   - Асинхронные запросы через async/await
   - Обработка загрузки и ошибок

2. **Методы визуализации:**
   - Компонентный подход React
   - Композиция компонентов для построения сложных интерфейсов
   - Условный рендеринг для отображения различных состояний

**3.4. Связи программы с другими программами**

1. **База данных PostgreSQL:**
   - Программа использует PostgreSQL для хранения всех данных
   - Связь осуществляется через JDBC драйвер и Spring Data JPA
   - Hibernate автоматически создает и обновляет схему базы данных

2. **Внешние API и сервисы:**
   - Программа может интегрироваться с ФГИС «Зерно» для автоматической отчетности (планируется)
   - Возможна интеграция с ERP-системами (1С, SAP) через REST API

3. **Бортовые модули техники:**
   - Программа принимает данные от аппаратных модулей, установленных на технике
   - Связь осуществляется через HTTP REST API
   - Модули передают регулярные пакеты данных с координатами, весом, уровнем заполненности

4. **Веб-браузер:**
   - Клиентская часть работает в веб-браузере пользователя
   - Использует стандартные веб-технологии (HTML, CSS, JavaScript)

### 4. Используемые технические средства

**4.1. Серверная часть:**

- **Тип ЭВМ:** Сервер с поддержкой Java 17 (x86_64 архитектура)
- **Минимальные требования:**
  - Процессор: 2 ядра, 2.0 GHz
  - Оперативная память: 2 GB
  - Свободное место на диске: 1 GB
  - Операционная система: Linux, Windows Server, macOS (для разработки)

- **База данных:**
  - PostgreSQL версии 16 или выше
  - Минимальные требования: 1 GB RAM, 500 MB дискового пространства

- **Сетевое оборудование:**
  - Сетевой интерфейс для подключения к локальной сети или интернету
  - Порт 8080 для HTTP-запросов (настраивается)

**4.2. Клиентская часть:**

- **Тип устройства:** Персональный компьютер, планшет или смартфон
- **Минимальные требования:**
  - Процессор: современный процессор с поддержкой JavaScript
  - Оперативная память: 512 MB
  - Операционная система: Windows 10+, macOS 10.14+, Linux (любое современное распределение)
  - Веб-браузер: Chrome 90+, Firefox 88+, Edge 90+, Safari 14+

**4.3. Сетевое оборудование:**

- Стабильное интернет-соединение для связи между клиентом и сервером
- Рекомендуемая скорость: не менее 1 Мбит/с для комфортной работы
- При работе в локальной сети: поддержка HTTP-протокола

**4.4. Дополнительное оборудование (для полной функциональности):**

- Бортовые аппаратные модули с датчиками:
  - Датчики веса зерна
  - Датчики уровня заполненности бункера
  - Датчики вибрации
  - Датчики уровня топлива
  - GPS/ГЛОНАСС модули для определения координат
  - Радиомодули LoRa для беспроводной связи (для автономной работы в поле)

### 5. Вызов и загрузка

**5.1. Способ вызова программы с соответствующего носителя данных**

**Серверная часть:**

1. **Через Docker Compose (рекомендуемый способ):**
   ```bash
   docker-compose up --build
   ```
   Программа запускается в контейнере Docker вместе с базой данных PostgreSQL.

2. **Локальный запуск:**
   ```bash
   ./gradlew bootRun
   ```
   или
   ```bash
   java -jar agrovzor-back-0.0.1-SNAPSHOT.jar
   ```

3. **Как системный сервис:**
   - Программа может быть настроена как systemd-сервис (Linux) или Windows Service
   - Автоматический запуск при загрузке операционной системы

**Клиентская часть:**

1. **Режим разработки:**
   ```bash
   npm run dev
   ```
   Приложение доступно по адресу http://localhost:3000

2. **Продакшн-сборка:**
   ```bash
   npm run build
   npm start
   ```
   Приложение доступно по адресу http://localhost:3000 (или настраиваемому порту)

3. **Через веб-браузер:**
   - Пользователь открывает веб-браузер
   - Переходит по URL сервера, на котором развернуто приложение
   - Приложение загружается автоматически

**5.2. Входные точки в программу**

**Серверная часть:**

- **Главный класс:** `ru.feryafox.agrovzorback.AgroVzorApplication`
- **Метод входа:** `public static void main(String[] args)`
- **Порт по умолчанию:** 8080 (настраивается в application.properties)

**Клиентская часть:**

- **Точка входа:** `src/app/layout.tsx` (корневой layout Next.js)
- **Главная страница:** `src/app/page.tsx`
- **URL по умолчанию:** http://localhost:3000

**5.3. Адреса загрузки, сведения об использовании оперативной памяти, объем программы**

**Серверная часть:**

- **Объем JAR-файла:** ~50-100 MB (в зависимости от зависимостей)
- **Использование оперативной памяти:**
  - Минимум: 512 MB
  - Рекомендуется: 1-2 GB
  - Максимум: настраивается через JVM параметры (-Xmx)
- **Адрес загрузки в памяти:** Управляется JVM автоматически

**Клиентская часть:**

- **Объем собранного приложения:** ~5-10 MB (после сборки и минификации)
- **Использование оперативной памяти браузером:**
  - Минимум: 100-200 MB
  - Типичное: 200-500 MB
  - Зависит от объема загружаемых данных

**База данных:**

- **Объем базы данных:** Зависит от объема хранимых данных
- **Типичный размер:** 100 MB - 10 GB (в зависимости от количества записей)
- **Использование оперативной памяти PostgreSQL:** Настраивается через postgresql.conf

**6.1.1. Данные от пользователя через веб-интерфейс:**

1. **Данные о предприятиях:**
   - Полное наименование предприятия (строка, обязательное)
   - Сокращенное наименование (строка, обязательное)
   - Формат: JSON через REST API
   - Предварительная подготовка: валидация на клиенте перед отправкой

2. **Данные о сотрудниках:**
   - Имя пользователя (username) - строка, уникальное
   - Фамилия, имя, отчество - строки
   - Связь с предприятием (UUID)
   - Формат: JSON через REST API

3. **Данные о технике:**
   - Тип техники (наименование)
   - Модель техники (наименование, связь с типом)
   - Единица техники (инвентарный номер, связь с моделью и предприятием)
   - Формат: JSON через REST API

4. **Данные о культурах:**
   - Вид культуры (наименование)
   - Сорт культуры (наименование, связь с видом)
   - Формат: JSON через REST API

5. **Данные о работах на поле:**
   - Связь с предприятием (UUID)
   - Связь с сортом культуры
   - Даты начала и окончания работ
   - Формат: JSON через REST API

6. **Данные о рабочих выездах:**
   - Связь с работой на поле
   - Связь с сотрудником
   - Связь с единицей техники
   - Даты начала и окончания выезда
   - Формат: JSON через REST API

**6.1.2. Телеметрические данные от бортовых модулей:**

1. **Регулярные пакеты данных (RegularPacket):**
   - UUID рабочего выезда
   - Вес зерна (число, килограммы, опционально)
   - Уровень заполненности бункера (число, проценты 0-100, опционально)
   - Координаты (широта, долгота)
   - Направление движения (число, градусы, опционально)
   - Временная метка (ISO 8601 формат)
   - Сырые данные (строка, опционально)
   - Формат: JSON через HTTP POST запрос
   - Частота передачи: каждые 5-10 секунд во время работы техники

2. **Данные вибрации (VibrationData):**
   - UUID рабочего выезда
   - Расположение датчика (строка, например "Выгрузной шнек")
   - Значения вибрации (строка с числами через запятую, например "12.5, 13.1, 12.8")
   - Временная метка
   - Формат: JSON через HTTP POST запрос

3. **Данные топлива (FuelData):**
   - UUID рабочего выезда
   - Уровень топлива (число, проценты 0-100)
   - Временная метка
   - Формат: JSON через HTTP POST запрос

**6.2. Формат, описание и способ кодирования входных данных**

**6.2.1. Формат данных:**

Все входные данные передаются в формате JSON (JavaScript Object Notation) с кодировкой UTF-8.

**6.2.2. Описание структуры данных:**

**Пример запроса создания предприятия:**
```json
{
  "fullName": "Общество с ограниченной ответственностью \"Агропромышленный комплекс\"",
  "shortName": "ООО \"АПК\""
}
```

**Пример регулярного пакета телеметрии:**
```json
{
  "workShiftId": "123e4567-e89b-12d3-a456-426614174000",
  "weight": 1250.5,
  "fillLevel": 75.3,
  "coordinate": {
    "latitude": 47.2357,
    "longitude": 39.7015
  },
  "direction": 45.0,
  "timestamp": "2024-01-15T10:30:00",
  "rawData": "дополнительные данные в произвольном формате"
}
```

**Пример данных вибрации:**
```json
{
  "workShiftId": "123e4567-e89b-12d3-a456-426614174000",
  "sensorLocation": "Выгрузной шнек",
  "vibrationValues": "12.5, 13.1, 12.8, 14.2, 13.5",
  "timestamp": "2024-01-15T10:30:00"
}
```

**6.2.3. Способ кодирования:**

- **Кодировка:** UTF-8
- **Тип контента:** application/json
- **Метод передачи:** HTTP POST/PUT запросы
- **Заголовки запроса:**
  - Content-Type: application/json
  - Accept: application/json

**6.2.4. Валидация входных данных:**

- Обязательные поля проверяются на наличие
- Типы данных проверяются на соответствие ожидаемым
- Числовые значения проверяются на диапазоны (например, проценты 0-100)
- UUID проверяются на корректность формата
- Даты проверяются на корректность формата ISO 8601

### 7. Выходные данные

**7.1. Характер и организация выходных данных**

**7.1.1. Данные, возвращаемые через REST API:**

1. **Справочные данные:**
   - Списки предприятий, сотрудников, техники, культур
   - Данные о конкретных объектах по идентификатору
   - Формат: JSON массивы или объекты

2. **Аналитические данные:**
   - Оценки потерь зерна (вес, процент, метод расчета)
   - Данные вибрации с флагом аномалии
   - Данные топлива с флагом подозрения на кражу
   - Индексы эффективности операторов
   - Результаты подбора команды

3. **Статусные данные:**
   - Коды HTTP-ответов (200 OK, 404 Not Found, 400 Bad Request, 500 Internal Server Error)
   - Сообщения об ошибках с описанием

**7.1.2. Данные, отображаемые в веб-интерфейсе:**

   - Списки предприятий, сотрудников, техники, работ

   - График вибрации
