# ПРОГРАММА И МЕТОДИКА ИСПЫТАНИЙ

## 1. ОБЪЕКТ ИСПЫТАНИЙ

**Наименование программы:** Экосистема агротехники в поле для учета и мониторинга зерна (AgroVzor)

**Область применения:** Автоматизация и мониторинг процесса трехзвенной уборки урожая зерновых культур в сельскохозяйственных предприятиях.

**Обозначение:** AgroVzor v1.0

## 2. ЦЕЛЬ ИСПЫТАНИЙ

Целью проведения испытаний является проверка соответствия программы требованиям, установленным в техническом задании, и подтверждение готовности программы к эксплуатации.

**Основные задачи испытаний:**
- Проверка функциональных характеристик программы
- Проверка требований к надежности
- Проверка требований к производительности
- Проверка требований к информационной и программной совместимости
- Проверка требований к безопасности
- Проверка соответствия программной документации

## 3. ТРЕБОВАНИЯ К ПРОГРАММЕ

В ходе испытаний подлежат проверке следующие требования, заданные в техническом задании:

### 3.1. Требования к функциональным характеристикам

**3.1.1. Управление организационной структурой:**
- Создание, редактирование, удаление предприятий
- Создание, редактирование, удаление сотрудников
- Управление допусками сотрудников к технике
- Учет типов, моделей и единиц техники
- Управление культурами (виды и сорта)

**3.1.2. Планирование и учет полевых работ:**
- Регистрация работ на полях
- Учет рабочих выездов механизаторов
- Планирование состава команды

**3.1.3. Сбор и обработка телеметрических данных:**
- Прием регулярных пакетов данных от бортовых модулей
- Сохранение данных при отсутствии связи
- Синхронизация данных при восстановлении связи

**3.1.4. Аналитика и мониторинг:**
- Расчет потерь зерна при уборке урожая
- Вибродиагностика техники с детекцией аномалий
- Мониторинг уровня топлива с выявлением кражи
- Расчет индекса эффективности механизаторов
- Автоматический подбор команды для полевых работ

**3.1.5. Визуализация и отчетность:**
- Отображение данных на интерактивных картах
- Построение графиков и диаграмм
- Формирование отчетов

**3.1.6. Временные характеристики:**
- Время отклика API: не более 500 мс
- Время обработки телеметрических пакетов: не более 100 мс
- Время расчета аналитических показателей: не более 2 секунд

### 3.2. Требования к надежности

- Коэффициент готовности системы: не менее 99%
- Время восстановления после отказа: не более 5 минут
- Контроль входной и выходной информации
- Защита от потери данных
- Обработка исключительных ситуаций

### 3.3. Требования к информационной и программной совместимости

- Формат данных: JSON (UTF-8)
- Протокол: HTTP/HTTPS REST API
- Языки программирования: Java 17, TypeScript 5
- Совместимость с указанными версиями программных средств

### 3.4. Требования к безопасности

- Защита от SQL-инъекций
- Защита от XSS-атак
- Валидация входных данных
- Логирование операций
- Использование HTTPS

## 7. МЕТОДЫ ИСПЫТАНИЙ

### 7.1. Метод проверки функциональных характеристик

**7.1.1. Проверка управления организационной структурой:**

**Тестовый пример 1: Создание предприятия**
- Входные данные: JSON с полным и кратким наименованием предприятия
- Действия: Отправка POST запроса на /api/companies
- Ожидаемый результат: Создание предприятия с присвоенным UUID, возврат данных в формате JSON
- Критерий приемки: Предприятие создано, UUID присвоен, данные возвращены корректно

**Тестовый пример 2: Получение списка предприятий**
- Входные данные: GET запрос на /api/companies
- Действия: Отправка GET запроса
- Ожидаемый результат: Возврат списка всех предприятий в формате JSON
- Критерий приемки: Список возвращен, формат данных корректный

**7.1.2. Проверка расчета потерь зерна:**

**Тестовый пример 3: Расчет потерь зерна**
- Входные данные: UUID рабочего выезда с телеметрическими данными
- Действия: Отправка POST запроса на /api/analytics/grain-loss/calculate/{workShiftId}
- Ожидаемый результат: Расчет потерь зерна (вес и процент), возврат данных в формате JSON
- Критерий приемки: Расчет выполнен корректно, значения в допустимых пределах

**7.1.3. Проверка вибродиагностики:**

**Тестовый пример 4: Детекция аномалии вибрации**
- Входные данные: Данные вибрации со значениями > 100
- Действия: Отправка POST запроса на /api/analytics/vibration
- Ожидаемый результат: Аномалия детектирована, флаг anomalyDetected = true
- Критерий приемки: Аномалия детектирована корректно

**7.1.4. Проверка мониторинга топлива:**

**Тестовый пример 5: Детекция кражи топлива**
- Входные данные: Данные топлива с падением уровня > 10%
- Действия: Отправка POST запроса на /api/analytics/fuel
- Ожидаемый результат: Кража детектирована, флаг theftDetected = true
- Критерий приемки: Кража детектирована корректно
